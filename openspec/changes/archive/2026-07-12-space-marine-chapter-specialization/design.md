## Context

`src/data/factions/space-marine-chapters.ts` currently defines a standalone `Faction` (`id: 'space-marine-chapters'`) with its own 19 units and 12 lettered upgrade groups (A–L), registered in `rulesDatabase.factions` alongside `space-marines.ts` (`id: 'space-marines'`, groups A–O). Every unit's `id` is baked at data-build time as `${factionId}.${nameToId(name)}` by the `unit()`/`faction()` helpers in `src/data/factions/helpers.ts`, and every list-scoped lookup in the app (`getFaction`, `getUnit`, `stores/lists.ts`, `BuilderView.vue`, `PrintView.vue`) resolves a single flat `Faction` object from one `factionId` string. The four chapters' point-cost rule modifiers exist only as unwired prose entries in that faction's `armyRules` (e.g. `armyRule('Blood Angels Chapter', 'Modifiers for Space Marine armies: Infantry get Furious for +10pts; Vehicles get Fast for +5pts.')`) — there is no option, calc.ts logic, or UI control that lets a user actually apply them.

Two existing mechanisms make this tractable without touching the core rules engine:
- `src/domain/calc.ts` and `src/domain/resolve.ts` are already faction-agnostic — every function takes a single, already-assembled `Faction` object and has no notion of "which faction" beyond that. Rule resolution (`resolve.ts`'s `createResolver`) indexes the global glossary plus `faction.armyRules`, and the calc functions (`applyUpgrades`, `findSection`, `isSectionAvailable`, `isInfantry`, etc.) all operate purely on `Faction.units`/`Faction.upgradeGroups`.
- The established "pay points for a rule" pattern is exactly `UpgradeOption { costDelta, effects: { addRules } }` (e.g. Space Marines group H's `{ label: 'Storm Shield (Tough(3))', cost: 5, adds: ['Tough(3)'] }`), used pervasively already. `Furious`, `Fast`, and `Scout` are already global-glossary rules (`src/data/glossary.ts`); `Aegis`, `Counter-Attack`, `Deathwing`, and `Wolf` are already defined as `armyRules` in `space-marine-chapters.ts`.

This means a chapter's bonuses can be implemented as ordinary upgrade options with no changes to `calc.ts`/`resolve.ts`, as long as the app assembles one merged `Faction` object (base Space Marines + chosen chapter) before handing it to those modules.

## Goals / Non-Goals

**Goals:**
- Chapters become an optional specialization of a Space Marines list, not a separate faction in the picker.
- Each chapter's bonuses are purchasable, per-unit upgrade options (matching "may take X for +Npts" wording), not automatic blanket effects.
- `calc.ts` and `resolve.ts` remain completely unaware of "chapters" — they keep consuming one flat `Faction`.
- A plain Space Marines list with no chapter behaves exactly as it does today (same units, same groups, same ids) — zero regression risk for the common case.
- Existing saved lists (localStorage) keep working: plain Space Marines lists unaffected; legacy `space-marine-chapters` lists are migrated to a Space Marines list with an inferred chapter.

**Non-Goals:**
- No general "sub-faction"/"variant" framework for other factions — this is scoped to Space Marines chapters only.
- No UI for switching a list's chapter after creation (matches how faction itself isn't changeable after creation today).
- No attempt to preserve exact upgrade-selection state across the legacy-data migration beyond best-effort id remapping — see Risks.

## Decisions

1. **Represent chapter on the list, not on the faction id.** Add `chapterId?: ChapterId` to `ArmyList` (`src/domain/list.ts`), keeping `factionId: 'space-marines'` unchanged. A new `ChapterId` union (`'blood-angels' | 'dark-angels' | 'grey-knights' | 'space-wolves'`) lives alongside the new chapter-assembly module.
   - *Alternative considered*: encode the chapter into a synthetic `factionId` like `'space-marines:blood-angels'`. Rejected — it would require every `getFaction`/`getUnit` call site to parse a compound id, and would make "faction" and "chapter" inseparable for the many places (roster header, print view) that want to show them as two distinct facts ("Space Marines (Blood Angels)").

2. **Assemble the effective faction dynamically, not as four pre-baked exports.** Add `src/data/chapters.ts` exporting `getEffectiveFaction(factionId: string, chapterId?: string): Faction | undefined`: for any `factionId !== 'space-marines'` or an absent `chapterId`, it returns the plain `getFaction(factionId)` result unchanged (identity passthrough, byte-for-byte the same object as today). For `'space-marines'` plus a `chapterId`, it returns a freshly merged `Faction` object: base `spaceMarines.units` plus that chapter's units, base `upgradeGroups` plus that chapter's (re-namespaced) upgrade groups plus a synthesized "Chapter Tactics" group, and base `armyRules` plus that chapter's `armyRules` (deduped by id, since `Narthecium`/`Tactics` already exist on both).
   - *Alternative considered*: four fully static pre-built `Faction` exports (`spaceMarinesBloodAngels`, etc.). Rejected — the datasets are small (≤19 extra units, a handful of groups), so the assembly cost is negligible, and a pure function is easier to unit-test and keeps the "no chapter" path trivially identical to today's data with no risk of the static export drifting from the base faction.

3. **Chapter units move into `space-marines.ts`'s id namespace, split by chapter.** Restructure `src/data/factions/space-marine-chapters.ts` to export four `{ units: UnitInput[], upgradeGroups: UpgradeGroup[] }`-shaped bundles (one per chapter), still built with the existing `unit()`/`group()`/`section()`/`option()` helpers but constructed with `factionId: 'space-marines'` (not `'space-marine-chapters'`), so every chapter unit's id becomes `space-marines.<slug>` — consistent with base units and resolvable through the same `getUnit('space-marines', id)` path once merged. Upgrade group ids are re-namespaced per chapter (e.g. Blood Angels' existing `A`–`D`/`G` become `ba-a`..`ba-g`, Dark Angels' `E`/`F` become `da-e`/`da-f`, etc.) so they don't collide with base Space Marines' `A`–`O` or with each other when a future faction assembly needs more than one chapter's groups loaded at once (it never does today, but namespacing costs nothing and avoids a subtle bug class).
   - *Alternative considered*: keep chapter units in a separate `'space-marine-chapters'`-id namespace and only merge `units`/`upgradeGroups` arrays at assembly time without renaming. Rejected — `getUnit(list.factionId, unitId)` is called with `list.factionId === 'space-marines'` throughout the app; a unit id still prefixed `space-marine-chapters.*` would never resolve through that path, forcing every call site to know about two id namespaces instead of one.

4. **Chapter Tactics options are synthesized per matching unit at assembly time, not authored per-unit in the source file.** In `chapters.ts`, for each chapter define a small declarative table of `{ appliesTo: (unit: UnitProfile) => boolean, label: string, cost: number, rule: string }` rows (e.g. Blood Angels: `{ appliesTo: isInfantryUnit, label: 'Furious', cost: 10, rule: 'Furious' }`, `{ appliesTo: isVehicleUnit, label: 'Fast', cost: 5, rule: 'Fast' }`; Dark Angels: `{ appliesTo: byName('Terminators'), label: 'Deathwing', cost: 20, rule: 'Deathwing' }`, `{ appliesTo: byName('Bike Squad'), label: 'Scout', cost: 10, rule: 'Scout' }`, `{ appliesTo: byName('Attack Bike'), label: 'Scout', cost: 5, rule: 'Scout' }`; Grey Knights: Infantry and Vehicle both get `Aegis` for 5pts; Space Wolves: `{ appliesTo: isHeroUnit, label: 'Wolf', cost: 30, rule: 'Wolf' }`, `{ appliesTo: isInfantryUnit, label: 'Counter-Attack', cost: 10, rule: 'Counter-Attack' }`). `isInfantryUnit`/`isVehicleUnit`/`isHeroUnit` reuse `src/domain/calc.ts`'s existing `isInfantry` and the same rule-id-presence pattern for Vehicle/Hero. For every unit (base + chapter) matching a row's `appliesTo`, the assembler appends one `UpgradeOption` (via the existing `option()` helper, `costDelta` + `effects.addRules`) into a per-chapter `'<chapter>-tactics'` group, and appends that group's id to the matching unit's (cloned) `upgradeGroups` list. Units are shallow-cloned before mutation so the base `spaceMarines` singleton is never mutated.
   - *Alternative considered*: model each bonus as a real unit-specific upgrade group hand-authored once per applicable base unit (e.g. add a `P` group to `Terminators` directly in `space-marines.ts`). Rejected — the bonus only applies when a chapter is selected, so baking it directly into the base faction's static groups would show "Deathwing (+20pts)" on Terminators even in chapter-less Space Marines lists, and would require four different hand-maintained copies of the same base units for each chapter's Tactics option. The declarative-table approach keeps `space-marines.ts` completely unchanged and expresses each chapter's rule as the single source of truth for that rule.

5. **`ArmyList.chapterId` threads through the store and UI exactly like `factionId` does today.** `createList(name, factionId, pointsCap, chapterId?)`; every list-scoped `getFaction(list.factionId)` call in `stores/lists.ts` (`sanitizeLinks`, `groupIsValid`, `combineUnits`, `attachToUnit`, `joinGroup`, `toggleUpgrade`) and in `BuilderView.vue`/`PrintView.vue` becomes `getEffectiveFaction(list.factionId, list.chapterId)`. `validateImported` accepts an optional `chapterId` field, validates it's one of the four known ids and only present when `factionId === 'space-marines'`, and resolves units/upgrades against the assembled effective faction.

6. **`CreateListDialog.vue` shows the chapter selector conditionally.** A `<select>` for chapter (`None` + the four chapters) renders only when the currently-selected faction is `'space-marines'`, defaulting to `None`; changing the faction away from Space Marines resets the chapter selection back to `None`. The `create` event payload gains an optional `chapterId`.

## Risks / Trade-offs

- [Risk] Re-namespacing chapter upgrade group ids (decision 3) changes every `selectedUpgrades` id a legacy saved list might reference for chapter units → Mitigation: bump `LIST_SCHEMA_VERSION` to 2 and add a one-time migration in `stores/lists.ts`'s `load()` that, for any list with `factionId === 'space-marine-chapters'`: (a) rewrites `factionId` to `'space-marines'`, (b) infers `chapterId` as the chapter owning the majority of the list's existing `unitId`s (a small static lookup table of the 19 legacy unit slugs → chapter, kept only in the migration code), (c) remaps each unit's `unitId` and each `selectedUpgrades` entry's group-letter prefix to the new namespaced ids for that inferred chapter, and (d) drops (with a console warning, not a thrown error) any unit entry whose old id belongs to a *different* chapter than the inferred majority, since it can't resolve against the new single-chapter roster. This is a best-effort, one-time migration for what is expected to be a small number of pre-existing lists (client-only localStorage, no server-side data).
- [Risk] A saved list's `updatedAt`/id/name are unaffected by migration, but its total point cost could change if a dropped minority-chapter unit contributed points → Mitigation: acceptable one-time data loss for an edge case (mixed-chapter lists were never a designed use case, just an accidental side effect of chapters being one flat faction); document it in the changelog/commit message.
- [Risk] Synthesizing upgrade options at read time (decision 2/4) means every `BuilderView`/`PrintView`/store call for a chapter list re-runs the assembly — if done naively per-render this could be wasteful → Mitigation: the dataset is tiny (≤19 units, ≤6 tactic rows per chapter); no memoization needed at this scale, but if profiling shows it matters, `getEffectiveFaction` can be wrapped in a `computed()` keyed on `(factionId, chapterId)` at the call sites that need it repeatedly (`BuilderView.vue`'s `faction` computed already does this pattern for the non-chapter case).
- [Risk] `isInfantryUnit`/`isVehicleUnit` predicates for Chapter Tactics eligibility must exactly match the rulebook's "Infantry"/"Vehicle" categorization already encoded via `isInfantry()` in `calc.ts` (Infantry = absence of Hero/Psyker/Monster/Vehicle) → Mitigation: reuse `calc.ts`'s exported `isInfantry` directly rather than re-deriving the predicate, and add `isVehicle`/`isHero` as trivial one-line siblings next to it (checking for the `Vehicle`/`Hero` rule id) so all three predicates live in one place and stay consistent.

## Migration Plan

1. Bump `LIST_SCHEMA_VERSION` from 1 to 2 in `src/domain/list.ts`.
2. Add the legacy-chapter migration step to `stores/lists.ts`'s `load()` (see Risks above), gated on `factionId === 'space-marine-chapters'`.
3. Deploy is a plain static-site rebuild (no server/data-store migration beyond the client-side `load()` step, consistent with `single-file-build`'s all-client-side architecture).
4. Rollback: revert the commit. Since the migration only *rewrites* `factionId`/`unitId`/`selectedUpgrades` strings in place (never deletes the underlying localStorage key), a rollback to the previous build would simply fail to resolve the already-migrated `factionId: 'space-marines'` chapter-tagged lists back to the old `'space-marine-chapters'` faction — acceptable given this is a hobby project without a formal rollback SLA; call out in the PR description if this ships to users with existing chapter lists.
