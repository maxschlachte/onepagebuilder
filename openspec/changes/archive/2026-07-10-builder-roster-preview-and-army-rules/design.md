## Context

This bundles five related but structurally distinct gaps found while reviewing the builder/print flow against what `rules-data` and `print-view` already promise:

1. The roster (`BuilderView.vue`'s left column) shows only `[size] · Qquality · cost` — not the unit's baseline equipment/rules — before it's added.
2. `EntryUpgradeControls.vue` renders an option's `label` as plain text; `RuleTooltip`/`RuleChips` (already used for a unit's *applied* special rules, weapon rules, and psychic-power hover text) are never used for upgrade options themselves.
3. Several upgrade options — starting with every faction's "Battle Standard" option (`{ label: 'Battle Standard', cost: 90 }`, no `effects`) — don't declare the rule they're named after as an `addRules`/`adds` effect, even though `rules-data`'s existing "Upgrade groups" requirement already says an option records its effects on special rules. Because of this, selecting "Battle Standard" today changes nothing about the unit's `specialRules`, so it never reaches `PrintView.vue`'s `reference` computed (which walks `eff.specialRules`) — silently violating the already-shipped "Self-sufficient rule reference" print-view requirement for any rule reached only through an upgrade rather than a baseline special rule.
4. `PrintView.vue`'s `reference` computed merges psychic powers into the same id set as special/army/weapon rules (`if (hasPsyker) for (const p of faction.value.psychicPowers) ids.add(p.id)`), so they render in one "Rule Reference" section instead of their own.
5. Four faction files already carry an army rule reading "You may deploy up to 10 models with this rule together to form a single unit" (`Conclave` in Sisters of Battle, `Warband` in Inquisition, `Beastmaster`/`Court` in Dark Eldar), but nothing in the app lets a user actually combine entries under it — only the unrelated, already-shipped "two copies of the same Infantry unit" combine (`ListUnit.combinedWith`, symmetric pairwise, same `unitId` only, exactly 2) exists.
6. The roster shows a `Hero` badge (`unit.isHero`) but no equivalent for a Psyker-only unit, even though `BuilderView.vue` already treats a bare Psyker exactly like a Hero for attach purposes (`isHeroOrPsyker`).

## Goals / Non-Goals

**Goals:**
- Make the roster show enough of a unit's stats to decide whether to add it, without leaving the builder.
- Make "an upgrade option grants rule X" a real, tooltip-visible, print-reference-visible fact, not just a label string — for every faction, not only the one option named in the request.
- Split Psychic Powers into their own printed section, same visual treatment as Rule Reference.
- Add builder/store/print support for the rulebook's "up to 10 models sharing a rule → one unit" mechanic, generalized from (not layered onto) the existing 2-copies-same-unit Infantry combine, since the two are triggered differently (shared rule vs. shared `unitId`) and capped differently (10 models vs. 2 entries).
- Give a Psyker-only roster unit the same at-a-glance badge a Hero unit already gets.

**Non-Goals:**
- No change to `applyUpgrades`, cost math, or the existing Infantry `combinedWith`/`joinedInfantryUnit` mechanics — those stay exactly as shipped.
- No attempt to model the *combat* effect of Conclave/Warband/Beastmaster/Court beyond what the rulebook actually says (display/cost as one unit) — same scope boundary the Infantry combine already drew.
- No hardcoded list of "these 4 rule ids are group-combinable" — derived from rule text so a 17th faction file authored the same way picks it up for free, mirroring `isInfantry`/`affectsAllModels`'s existing derived-not-tagged precedent.
- No redesign of the tooltip/popover visual system — reuse `RuleTooltip`/`RuleChips`/`EquipmentList` as-is.

## Decisions

**1. Roster stat preview is a hover/focus popover reusing `EquipmentList` + `RuleChips` against the unit's *baseline* profile.**
No upgrades are selected yet, so the preview shows `unit.equipment`/`unit.specialRules` directly (no `applyUpgrades` call needed) — a new small wrapper component (`RosterUnitPreview.vue`) renders the same equipment-list/rule-chip markup `PrintUnitStats`/the builder's "Selected units" panel already use, inside a `RuleTooltip`-style CSS-only hover popover (`group-hover`/`group-focus`, no JS state), anchored on the roster row. Keyboard/touch users get it via `tabindex` + `:focus` the same way `RuleTooltip` already does.
Alternative considered: a click-to-expand `<details>` row (already used for the builder's per-unit "Psychic powers" disclosure). Rejected as the primary mechanism — hovering the roster to compare units before adding is a lighter-weight, more frequent interaction than the psychic-power disclosure's "read once after adding" use — but the roster row's existing text stays visible either way, so this is additive, not a replacement of the current one-line summary.

**2. An upgrade option's rule-granting effect is authored the same way every other option effect already is: `adds: ['Rule Name']` in faction data, parsed by the existing `parseRule`/`nameToId` pipeline.**
`armyRule('Battle Standard', …)`'s id is `nameToId('Battle Standard')`; an option declaring `adds: ['Battle Standard']` produces `effects.addRules = [{ ruleId: 'battle-standard' }]` — the exact id the army-rule entry already registers under in `createResolver`'s index (which already merges `faction.armyRules` alongside the glossary). No new field, parser, or resolver change — purely closing a data gap using machinery that already exists for every other rule reference in the codebase. This is audited and fixed across all 16 faction files (task list enumerates every option whose label names a real special/army rule but has no matching `addRules` entry today), not just "Battle Standard".
Alternative considered: a new dedicated `grantsArmyRule?: string` field on `UpgradeOption`, kept separate from `addRules`. Rejected — `addRules`/`RuleRef` already resolves against both the glossary and `faction.armyRules` uniformly (see `resolve.ts`), so a parallel field would just be two representations of the same fact with no behavioral difference, and would need its own resolver branch.

**3. Upgrade-option tooltips render only when the option's label textually *is* one of its granted rules (whole label, or the text before a trailing parenthetical) — not a chip appended after every label.**
Many labels already spell the rule out redundantly in parentheses ("Jump Pack (Deep Strike, Flying)", "Terminator Armor (Armored, Deep Strike)") — appending `RuleChips` there would duplicate text. So `EntryUpgradeControls.vue` checks, per option: for each `addRules` entry, resolve its name; if it equals the full label (`"Battle Standard"`) or the label's prefix before `" ("`, wrap that text in a `RuleTooltip` instead of a plain `<span>`. Labels with no such match (the common "descriptive name + parenthetical rule list" shape) keep today's plain-text rendering — unchanged visually, since the parenthetical already states the rule name in full, which is what a reader needs even without a tooltip there.
Alternative considered: always show the option's granted rules as trailing `RuleChips`, regardless of label shape. Rejected as visually noisy for the common case (would double every "(Rule1, Rule2)" suffix into "(Rule1, Rule2) Rule1, Rule2" via chips) for no readability gain over the existing parenthetical text.

**4. Print view splits `reference` into two independently-computed, independently-rendered sections by walking the same ids into two buckets instead of one.**
`PrintView.vue`'s existing single `ids: Set<string>` walk stays for special/army/weapon rules; psychic-power ids move to a second `Set` populated the same way (`hasPsyker` gate, `faction.value.psychicPowers`), each resolved and rendered through the same `dl`/`dt`/`dd` markup, in a second `<section>` titled "Psychic Powers" placed after "Rule Reference". No resolver change — `resolve()` already treats both id spaces identically; only which section a resolved id's markup lands in changes.

**5. Group-deployment combine is a new, separate mechanism (`ListUnit.groupId?: string` + a rule-derived eligibility check), not a generalization of `combinedWith`.**
`combinedWith` is a symmetric pairwise reference that assumes exactly 2 entries of the *same* `unitId` (Infantry's own wording: "two copies of the same Infantry unit"); Conclave/Warband/etc. allow *up to 10*, *possibly-different* unit types, keyed by a shared rule instead of a shared `unitId`. Forcing this into the pairwise field would mean either a linked list of pairwise links (awkward "walk the chain" logic for a 10-member group) or overloading the field's meaning. Instead:
- `groupId?: string` (a fresh id, e.g. from the same `uid()` used elsewhere) is set identically on every member of a group; `list.units.filter(u => u.groupId === lu.groupId)` finds the rest.
- `groupDeployRuleIds(faction): Set<string>` derives eligible rule ids from `faction.armyRules` text matching `/deploy up to (\d+) models .* (?:single|one) unit/i`, extracting the cap (currently 10 everywhere it appears, but read from the text rather than hardcoded).
- Two entries can join the same group only if their profiles' special rules share at least one common id from that set (a Warband model can't join a Conclave group even though both match the generic pattern) — the group "remembers" its anchor rule implicitly via its first member; joining validates the new member shares that rule with the existing member(s).
- The rulebook caps *models*, not *entries* — joining is rejected if the sum of joining members' `profile.size` would exceed the rule's extracted cap, not merely the entry count (matters if a future unit in this rule family has size > 1; today's four factions' members are all size 1, so this only matters for correctness/future-proofing, not current behavior).
- Store actions mirror the Infantry ones' shape (`joinGroup(listId, instanceId, groupMemberInstanceId)`, `leaveGroup(listId, instanceId)`), and get the same cleanup-on-remove, remap-on-duplicate, and drop-invalid-on-load treatment `combinedWith`/`joinedInfantryUnit` already have (extended to cover `groupId`, e.g. remap to a fresh id per group on duplicate so the copy's group doesn't alias the source list's).
Alternative considered: reuse `combinedWith` as an array (`combinedWith?: string[]`) covering both cases. Rejected — the Infantry rule's "must be bought for both" whole-unit-upgrade synchronization (design of `infantry-combine-and-attach`) has no equivalent here (group members are typically different unit types with unrelated upgrade groups, so there's nothing to synchronize), and conflating the two would force every `combinedWith` reader to handle an N-member case that Infantry itself never needs.

**6. Group aggregation for display/print is a new `groupEffectiveUnit(members)` function, not a reuse of `combinedEffectiveUnit`.**
`combinedEffectiveUnit(a, b)` returns a single `profile` (both entries share the same one) for display — meaningless for a group of different unit types. `groupEffectiveUnit` instead returns a small roster (`{ profile: UnitProfile; count: number }[]`, since a group could in theory hold more than one entry of the same member unit) alongside summed size/cost and deduped equipment/special rules (same merge-by-key logic as `combinedEffectiveUnit`, which doesn't actually depend on same-profile beyond the return shape). Builder/print rendering shows the member roster (name × count per distinct member) where a combined-Infantry box shows one profile name.

**7. Psyker roster badge reuses the existing `isHeroOrPsyker`-shaped check, but only when the unit isn't already Hero-badged.**
`unit.isHero ? 'Hero' : unit.specialRules.some(r => r.ruleId === 'psyker') ? 'Psyker' : null` — a unit that's both (several are, e.g. Farseer, Sorcerer's chapter variants) keeps showing just `Hero`, since Hero already implies the same attach-eligibility the Psyker badge would otherwise communicate, and double-badging every Hero-Psyker unit adds noise without new information.

## Risks / Trade-offs

- **[Risk]** The `addRules` audit across 16 faction files is manual/pattern-matched (an option's label "looks like" a rule name), so a genuine miss is possible. → **Mitigation**: add a database-wide audit test (matching this codebase's established `melee-weapon-audit`/`infantry-audit` pattern) asserting every option whose label exactly equals an `armyRules` entry's name declares that rule as an `addRules` effect — turns "did we miss one" into a CI-enforced invariant rather than a one-time manual pass.
- **[Risk]** The group-deployment rule-text regex (`/deploy up to (\d+) models .* single unit/i`) is a heuristic, like `affectsAllModels`'s title-word match — a future army rule phrased differently wouldn't be picked up. → **Mitigation**: same audit-test pattern, asserting the four known ids (`conclave`, `warband`, `beastmaster`, `court`) are all detected today, so a regex change that stops matching them fails the build immediately.
- **[Risk]** Group combine's cap-by-model-count (not by entry count) is currently unobservable in the four existing factions (all members are size 1), so a size-count bug there wouldn't be caught by manual testing against current data. → **Mitigation**: unit-test the cap check directly against a synthetic multi-model profile rather than relying on faction data to exercise it.
- **[Trade-off]** The upgrade-option tooltip only fires on an exact label/prefix match (decision 3), so an option whose label paraphrases its granted rule differently (e.g. a future "Banner"-shaped label where the rule name isn't the label's literal prefix) won't get a tooltip on the label itself — it still gets the tooltip via the existing on-unit special-rules display once selected, so this is a missed convenience, not a missed rule.

## Migration Plan

Additive only. `ListUnit.groupId` is a new optional field — existing saved lists deserialize with it `undefined`, same as `combinedWith`/`joinedInfantryUnit` before it; no `LIST_SCHEMA_VERSION` bump. Faction-data `addRules` additions only add effects to options that previously had none relevant to this fix — they don't change any option's cost, label, or existing equipment effects.

## Open Questions

- Should a group-deployment group show a "Split all" / per-member "Leave" control, or only per-member leave (mirroring Infantry's pair-only split)? Leaning: per-member leave only (a group can shrink one at a time down to a single remaining entry, which then simply has no `groupId` peers left and renders normally) — simpler store action, no special "disband whole group" case to test. Confirm during implementation once the store action is written.
