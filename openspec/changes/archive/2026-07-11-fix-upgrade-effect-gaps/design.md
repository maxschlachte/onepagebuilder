## Context

Two independent bug classes were found while reviewing `src/data/factions/dark-eldar.ts`, then confirmed/scoped against the rest of the dataset:

1. **Wrong removal semantics for multi-copy-on-one-model weapons.** `UpgradeOption.effects.removeOneEquipment` (see `src/domain/calc.ts:341-348`) decrements an equipment entry's `unitCount` (how many *models* carry it), defaulting to `unit.size` when unset. Dark Eldar's Ravager (`size: 1`) carries `customWeapon('Disintegrator Cannons', {...}, { count: 3 })` — three physical copies on the one model. Because `unitCount` is unset, it defaults to `unit.size = 1`, so a single `removeOneEquipment` selection computes `remaining = 1 - 1 = 0` and deletes the entire 3-copy entry, not one copy of it. Verified directly against `applyUpgrades`: selecting Group L's Dark Lance option once removes all 3 Disintegrator Cannons and adds only 1 Dark Lance.

   This exact shape (single model, `count > 1` weapon, a "replace one/any" upgrade) already exists elsewhere in the dataset — Deff Dred's `4x Light Powerfists`, Wraithlord's `2x Shotgun`, Wraithknight's `2x Wraithcannons`, Voidweaver/Starweaver's `2x Shuriken Cannon` — and in every one of those cases the option is addition-only (no `removeEquipment`/`removeOneEquipment` at all), per explicit precedent documented in two archived design docs (`openspec/changes/archive/2026-07-08-effective-equipment-display/design.md` and `.../2026-07-08-show-weapon-counts/design.md`, which lists this exact case as a Non-Goal: *"Per-model weapon-copy counts (the existing `count` field and its own 'additions only' vehicle edge cases) — untouched"*). Dark Eldar's Group L is the only place in the dataset that violates this. Its own inline comment claims the Ravager is protected by a "label mismatch" — it isn't; both Raider's and Ravager's entries resolve to the identical equipment key (`disintegrator-cannon`) via `equipmentKeyOf`'s depluralization, so the comment is simply incorrect.

2. **Missing rule effects.** Several upgrade options have a `label` naming a special rule in parentheses (e.g. `'Night Shields (Stealth)'`, `'Chain-Snares (Impact(+D3))'`) but no `adds:` field and no equipment carrying that rule — so selecting the option changes `costDelta` and nothing else. `helpers.ts`'s `parseRule` already handles delta-style params like `+D3`/`+D6` (any token containing a digit that isn't a bare integer is kept verbatim as `ref.param`), so `adds: ['Impact(+D3)']` is already fully supported by the existing engine; these are pure data omissions, not missing functionality.

3. **Weapon stats hand-typed into option labels, invisible until equipped.** Every faction file authors weapon-adding option labels like `'Dark Lance (36”, A6x)'` — the stat parenthetical is free text, typed independently of the `customWeapon('Dark Lance', { range: 36, attacks: '6', ... })` profile passed to `addEquipment` two lines later. Nothing checks the two agree; a typo in either place silently diverges from the other. `EntryUpgradeControls.vue` (the upgrade picker) renders `opt.label` verbatim, so today a weapon's stats are visible before selection *only* because they happen to be baked into the label text — an option authored without the parenthetical (there are several, e.g. any option that only reuses a global-table weapon by name) shows no stats at all until equipped. `EquipmentList.vue`, by contrast, already computes `range`/`attacks` from the structured `Weapon` profile at render time (`profile(e)` at `EquipmentList.vue:11-15`) and shows rules via `RuleChips` — this is the pattern to extend to the picker, not the hand-typed-label pattern to keep.

## Goals / Non-Goals

**Goals:**
- Ravager (and any other confirmed single-model/multi-copy unit sharing an upgrade group with a `removeOneEquipment` option) gets addition-only treatment, matching the established pattern.
- Every upgrade option whose label names a rule actually grants that rule via `adds`, across every faction file where this was confirmed missing.
- Correct the stale/incorrect inline comment on Dark Eldar Group L.
- A weapon-adding option's range/attacks/rules are visible next to it in the upgrade picker before selection, computed from its own `addEquipment` data — and its label can never diverge from that data because it no longer encodes it independently.
- Capture all three fixed behaviors as spec scenarios under the existing `rules-data`/`army-builder-ui` capabilities so future faction-data authoring has this documented, not just tribal knowledge in archived design docs.

**Non-Goals:**
- No change to `src/domain/calc.ts` or `helpers.ts` — the engine already does the right thing given correct data; only the faction data files and one display component change.
- Not re-litigating `selection: 'any'` sections that can only ever be picked once (the toggle-based selection store) — that's an existing, accepted, documented limitation (`show-weapon-counts` design doc Non-Goals), not part of this change.
- Not changing the *visual* tooltip mechanism for rule-granting labels (`labelTooltip` in `EntryUpgradeControls.vue`, decision 3 of `builder-roster-preview-and-army-rules`) — that logic is untouched; this change is additive alongside it (a new stats/rule-chip block, not a replacement of the existing label-tooltip rendering).
- Not rewording any option's descriptive text (e.g. "Blast Pistol", "Medium CCW") — only the embedded stat parenthetical is removed; the weapon/rule *names* an author chose stay exactly as printed.

## Decisions

**Fix Group L via addition-only, not by trying to make `removeOneEquipment` copy-aware.** Making `removeOneEquipment` distinguish "decrement model-count" from "decrement weapon-copy-count" would be a domain-engine change affecting all 16 factions' data semantics, for a pattern that occurs in exactly one place. The existing, tested convention (confirmed in 4 other units already) is addition-only for this shape; follow it rather than invent a new mechanism.

**Fix missing `adds` case-by-case, not with a lint rule.** The `2026-07-08-effective-equipment-display` design doc explicitly rejected blind automation for rule-vs-equipment classification ("audited case-by-case... only options whose rulebook entry itself represents picking up a distinct weapon get equipment effects") for the same reason a mechanical fix here would be risky: a parenthetical can be a weapon profile, an already-covered equipment rule, a deliberate no-op (e.g. Dark Eldar's `'Pistol (Poison)', cost: 0` duplicate-baseline option), or a genuine gap — telling these apart needs judgment per option, not a regex.

**Add two scenarios to `rules-data`'s existing "Upgrade groups" requirement rather than opening a new capability.** Both fixes are conformance fixes to behavior already implied by that requirement's opening sentence ("options record... effects on the unit's equipment and special rules"); the gap is that the spec never made the multi-copy exclusion or the rule-label case explicit, which is exactly what let both bugs go unnoticed. No new capability is introduced.

**Extract `EquipmentList.vue`'s `profile()` formatter into a shared module and reuse it (plus `RuleChips`) in `EntryUpgradeControls.vue`.** Both components need "given a `Weapon`, render `(36”, A6x)`-equivalent text and its rule chips" — `EquipmentList.vue:11-15` already implements this correctly against the structured profile. Duplicating the logic in the picker would just recreate the same drift risk this change exists to close (a second hand-maintained formatter that could disagree with the first). A small shared function (e.g. `formatWeaponProfile(weapon: Weapon): string` in a new `src/domain/format.ts`, or co-located in `calc.ts` next to the other display-adjacent helpers) is imported by both.

**`EntryUpgradeControls.vue` renders stats/rule-chips per weapon in `option.effects.addEquipment`, appended after the (now stats-free) label — not by parsing the label text.** For a single-weapon option this is one `(range, attacks)` + rule-chip block next to the label, matching `EquipmentList.vue`'s presentation. For a multi-weapon option (e.g. an option whose `addEquipment` adds both a pistol and a CCW), each entry gets its own bracketed stats immediately after its own name-fragment is unnecessary to reconstruct — instead the whole label is followed by each added weapon's stats in the *same order* `addEquipment` lists them, since option labels for multi-weapon adds already name the weapons in that same order (e.g. `'Blast Pistol, Medium CCW'` naming two `addEquipment` entries in the same sequence). This keeps the rendering logic simple (iterate `addEquipment`, skip entries with no `.weapon`) without needing to correlate substrings of the label to specific array entries. Options that grant equipment without an interesting profile (e.g. `gear()` entries with no `.weapon`) render no stats block, same as `EquipmentList.vue` already does for non-weapon equipment.

**Author convention: `OptionInput.label` names the weapon/rule, never its profile; a database-wide audit test enforces it.** Reusing the `looksLikeProfile` heuristic already in `helpers.ts` (`equipmentKeyOf`: a parenthetical containing a quoted range or an `A<digits/D>` attacks token), a new test walks every option's label across all 16 faction files and fails if any parenthetical matches that shape — turning "don't hand-type stats" from a convention into a CI-enforced invariant, the same pattern `weapon-count-audit.test.ts` already established for the equipment-count fixes. The 209 labels found by today's one-off audit become the fix list; the test then prevents the pattern from creeping back in.

## Risks / Trade-offs

- [Reclassifying an option as addition-only changes its selected effect for any list someone has already built with that option picked] → Dark Eldar's Group L currently *destroys* the unit's weapons when picked; addition-only is strictly more correct and more generous (keeps existing weapons, adds the new one), so no existing list gets worse, only more accurate.
- [Judgment calls on which parenthetical rule-name gaps are genuine vs. deliberate no-ops could misclassify a rare case] → Each fix keeps or adds an inline comment explaining the reasoning, consistent with the file's existing commenting convention, so a reviewer can spot-check the judgment call.
- [The multi-weapon label-ordering assumption (decision above) breaks if some option names weapons in a different order than its `addEquipment` array, or names a weapon its `addEquipment` doesn't carry a profile for] → Mitigated by the same audit test: it can additionally assert, for any option whose label lists N comma-separated weapon names, that `addEquipment` contains at least N entries with a `.weapon`; a mismatch fails the test instead of silently rendering stats next to the wrong name.
- [209 label edits across 16 files is a large mechanical change, similar in shape to the `show-weapon-counts` rollout] → Mitigated the same way that change was: one task per faction file for tracked, incremental progress, full test suite run after each file.
